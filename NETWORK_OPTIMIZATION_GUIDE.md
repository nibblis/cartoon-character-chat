# Оптимизация сети и Оффлайн-режим

Этот документ описывает стратегии минимизации трафика и обеспечения работоспособности приложения в условиях отсутствия сети.

---

## 1. Оптимизация трафика

В проекте реализованы следующие механизмы:
*   **Gzip Compression:** Использование заголовка `Accept-Encoding: gzip` позволяет сжимать JSON-ответы сервера.
*   **OkHttp Cache:** Настроен дисковый кэш (10 MiB) для хранения ответов. Это снижает количество повторных запросов (Latency) и экономит заряд батареи.
*   **Conditional Requests:** Использование `ETag` и `Last-Modified` позволяет серверу возвращать `304 Not Modified`, если данные не изменились, исключая передачу тела ответа.

---

## 2. Концепция Offline-first (Single Source of Truth)

Для обеспечения бесшовного UX при потере связи используется архитектурный паттерн **SSOT** с использованием Room:

### Алгоритм работы:
1.  **UI** подписывается только на данные из **Локальной БД (Room)** через Flow.
2.  **Repository** при запросе данных:
    *   Сначала отдает данные из БД (мгновенное отображение).
    *   Запускает сетевой запрос в фоне.
    *   При успехе — обновляет данные в БД.
    *   UI автоматически получает обновление через Flow.

### Обработка исходящих данных (Sync):
*   Сообщение сначала сохраняется в БД со статусом `SENDING`.
*   Запускается фоновая задача (через WorkManager для надежности).
*   При появлении сети данные отправляются на сервер, статус в БД меняется на `SENT`.

---

## 3. Метрики успеха
*   **Data Usage:** Снижение объема входящего трафика на 60% благодаря Gzip и кэшированию.
*   **First Contentful Paint:** Отображение списка чатов за <100мс (из кэша БД) вместо ожидания ответа сервера (1-2 сек).
*   **Reliability:** 100% сохранение сообщений пользователя даже при полном отсутствии интернета в момент нажатия кнопки "Отправить".
