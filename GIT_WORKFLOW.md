### Обзор рабочего процесса c Git

Этот документ кратко описывает основные практики работы с Git, принятые в этом проекте, на основе стратегии **GitFlow**.

---

### 1. Стратегия ветвления: GitFlow

Мы используем **GitFlow** — надежную модель ветвления, которая отлично подходит для проектов с запланированными релизами и необходимостью поддержки нескольких версий.

**Основные ветки:**
-   `main` (или `master`): Хранит только стабильный, готовый к релизу в продакшн код. Каждый коммит в `main` — это новая версия.
-   `develop`: Основная ветка для разработки. В нее сливаются все готовые `feature`-ветки.

**Вспомогательные ветки:**
-   `feature/*`: Для разработки нового функционала. Создаются от `develop` и вливаются обратно в `develop`.
-   `release/*`: Для подготовки нового релиза. Создаются от `develop`. Позволяют отделить стабилизацию релиза от разработки новых фич.
-   `hotfix/*`: Для срочных исправлений в `main`. Создаются от `main` и вливаются как в `main`, так и в `develop`.

**Почему GitFlow?**
-   **Стабильность:** Строгое разделение `main` и `develop` гарантирует, что в релизной ветке всегда находится стабильный код.
-   **Параллельная разработка:** Команда может работать над новыми фичами в `develop`, пока одна группа занимается стабилизацией релиза в `release`-ветке.
-   **Четкий релизный цикл:** Идеально подходит для мобильных приложений, где релизы происходят по графику (например, раз в две недели).

---

### 2. Правила оформления Pull Request (PR)

-   **Заголовок:** `тип(область): описание` (например, `feat(profile): add avatar upload`).
-   **Описание:** Что и зачем сделано.
-   **Шаги для проверки:** Инструкции для рецензента.

---

### 3. Релизный цикл и Hotfix'ы в GitFlow

-   **Подготовка релиза (v1.2.0):**
    1.  Создается ветка `release/v1.2.0` от `develop`.
    2.  В ней происходят только исправления багов и подготовка к релизу.
    3.  Когда ветка готова, она вливается **и в `main`, и в `develop`**.
    4.  На коммите слияния в `main` ставится тег `v1.2.0`.

-   **Hotfix (Срочное исправление в v1.2.0):**
    1.  Создается ветка `hotfix/fix-login-crash` от `main`.
    2.  Вносится исправление.
    3.  Ветка вливается **и в `main`, и в `develop`**, чтобы исправление попало и в следующий релиз.
    4.  На коммите слияния в `main` ставится тег `v1.2.1`.

---

### 4. История и базовые команды

-   **Единообразие истории:** Используем Conventional Commits для чистоты истории.
-   **Базовый флоу:** `git clone` -> `git checkout develop` -> `git checkout -b feature/my-new-feature` -> `git add .` -> `git commit` -> `git push` -> создание PR в `develop`.
-   **Решение конфликтов:** Конфликты решаются в `feature`-ветке путем слияния с `develop` (`git merge develop`).
