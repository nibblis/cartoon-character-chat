# Анализ Garbage Collection (GC) в Android

Работа с логами ART (Android Runtime), распознавания типов сборщиков мусора и оптимизации производительности приложения.

---

## 1. Распознавание типа GC по логам

В современных версиях Android (начиная с Android 14+) основным сборщиком является **CMC (Concurrent Mark-Compact)**.

**Пример из логов приложения:**
```text
Using CollectorTypeCMC GC.
```

### Сравнение основных типов:
*   **CMC (Concurrent Mark-Compact):** Современный коллектор. Уплотняет кучу параллельно с работой приложения, минимизируя паузы.
*   **CC (Concurrent Copying):** Использовался ранее. Разделяет кучу на регионы и копирует живые объекты.
*   **GSS (Generational Semi-Space):** Часто встречается в фоновых процессах для быстрой очистки "молодых" объектов.

---

## 2. Анализ пауз STW (Stop-The-World)

Паузы STW — это моменты, когда выполнение кода приложения полностью останавливается для работы GC. В логах их можно вычислить по косвенным признакам или прямым метрикам `Davey`.

**Разбор лога производительности:**
```text
Skipped 49 frames! The application may be doing too much work on its main thread.
Davey! duration=867ms; Flags=0, ... DrawStart=29826117801966, FrameCompleted=29826163441133
```

**Анализ:**
*   **Skipped 49 frames:** Это критическая потеря плавности. При частоте 60fps это ~800мс задержки.
*   **Davey!:** Системное уведомление о "длинном" кадре (jank). 
*   **Связь с GC:** Если такие логи появляются в момент интенсивного создания объектов (например, загрузка 1000 сообщений чата без Sequences), это указывает на **GC_FOR_ALLOC** — блокирующую сборку мусора, вызванную нехваткой памяти для новой аллокации.

## 3. Рекомендации по тюнингу на Android

Для снижения нагрузки на GC в данном проекте применены следующие меры:

1.  **Использование Sequences (`asSequence`):**
    *   *Проблема:* `list.filter{}.map{}` создает промежуточные списки.
    *   *Тюнинг:* `asSequence()` обрабатывает элементы по одному «в трубе», не забивая кучу временными объектами.

2.  **Избегание Boxing (упаковки примитивов):**
    *   *Проблема:* `List<Int>` хранит объекты `Integer`, что создает нагрузку на GC.
    *   *Тюнинг:* Использование `sumOf`, `forEach` по массивам примитивов или специализированных коллекций (например, `SparseIntArray` в Android).

3.  **Оптимизация аллокаций в UI:**
    *   Не создавать объекты внутри `onDraw` (в Custom Views) или внутри Compose-функций без `remember`.
    *   Использование `String.builder` или шаблонов строк вместо конкатенации в циклах.

4.  **Управление размером кучи:**
    *   Для чатов с большим количеством медиа-контента уместно использование `android:largeHeap="true"` в `AndroidManifest.xml`, но только как дополнение к оптимизации кода.
